方案概述
🎯

目标

将 Cloudflare 边缘节点产生的 HTTP 访问日志（Logpush JSON 格式）自动转换为 CDN 合作方要求的自定义格式，并近实时推送到对方日志接收服务器。

🏗️

技术栈

全部基于 Cloudflare 原生服务：Logpush + R2 + Queues + Workers。无需服务器，无需运维，系统自动扩缩容。

📋

目标格式

145 字段 \u0001 分隔纯文本格式，HTTP Body 需 Gzip 压缩，URL 签名采用 MD5 鉴权方式。

⚡

端到端延迟

从请求产生到日志送达约 70 秒，通常远优于合作方 SLA 要求（典型要求为 3 分钟内）。

核心设计决策

流式 Gzip 解压

原始日志文件通过 DecompressionStream + ReadableStream 流式处理。Worker 无需将整个文件载入内存，逐行解压转换，内存占用恒定，不受文件大小影响。这是 Workers 128 MB 内存限制下的必要设计。

两级 Queue 架构

parse-queue 负责 R2 → 格式转换；send-queue 负责 R2 → HTTP 推送。两级分离实现独立重试策略：解析失败重试 2 次，发送失败重试 5 次。每个 Queue 均配置独立死信队列（DLQ）用于可观测性与人工恢复。

重复投递控制

Queues 采用 at-least-once 语义。Sender 同 batch 内逐条 await 串行发送；send-queue 的 max_concurrency 不设置，由 Queue autoscaler 自动扩缩多个并发 invocation。确认发送成功后写入 .done 标记，后续重复消息若看到标记或发现 R2 临时文件已删除，则静默 ack——这缩窄了 Queue at-least-once 重复投递的窗口。

时间戳兼容

Parser 自动识别 EdgeStartTimestamp 的三种格式：Unix 秒整数、Unix 毫秒整数、RFC 3339 字符串。推荐 Logpush 配置 timestamp_format = unix 以保证一致性。

CDN 节点国家码映射

目标格式要求的是 CDN 节点所在国家（非客户端国家）。EdgeColoCode（IATA 机场三字码，如 NRT = 东京）通过内置映射表转换为 ISO 3166-1 alpha-2 国家码，映射表按当前活跃的 Cloudflare POP 机场码持续维护。若 EdgeColoCode 为空或未命中映射表，则回退返回 SG。

请求时长字段（#6 request_time）

字段 #6 使用 EdgeEndTimestamp - EdgeStartTimestamp 计算，单位为秒并保留毫秒精度。该值表示 Cloudflare edge 收到请求到完成向客户端发送响应的时间。

MD5 鉴权

Workers 的 SubtleCrypto API 不支持 MD5 算法，因此 src/index.js 内嵌了纯 JS 实现的 RFC 1321 MD5。鉴权 URL 签名格式为：MD5({uri}-{ts}-{rand}-{privateKey})。

响应内容长度字段（#21 sent_http_content_length）

http_requests 数据集没有直接暴露 Content-Length 字段。字段 #21 从 ResponseHeaders['content-length'] 取值，需通过 API 配置 Logpush Custom Fields（见部署步骤 6b）。未配置或响应头中不含该字段时，统一返回 -。

Content-Length 的 Custom Fields 配置是可选项。不配置时字段 #21 固定输出 -，与该格式中其他无数据字段的处理方式一致。

各阶段触发机制详解

第一阶段 — Logpush → R2  时间驱动，约每 60 秒一次

Logpush 将过去约 60 秒内所有边缘节点产生的日志汇聚打包，写入一个 .log.gz 文件到 R2。这是整个流水线中唯一的时间驱动触发点，后续所有阶段均为事件驱动。

无论流量大小，约 60 秒触发一次：低流量时文件可能只有 3 条日志；流量峰值时可能有几十万条日志，文件几百 MB。

第二阶段 — R2 → parse-queue  事件驱动，1 个文件 = 1 条消息

R2 每写入一个新文件，立即触发 Event Notification，向 parse-queue 写入一条消息。消息只包含 R2 对象路径（数十字节），彻底规避 Queue 128KB 消息大小限制。

R2 写入完成到 Queue 消息可见的延迟：通常 2–5 秒。

第三阶段 — Parser Worker  事件驱动，按 max_batch_size / max_batch_timeout 批量聚合

满足以下任一条件时 Worker 被唤醒：

  • max_batch_size = 3：积累到 3 条消息 → 立即触发
  • max_batch_timeout = 10s：不足 3 条但已等待 10 秒 → 同样触发

对每个文件，Parser 每处理 1000 条就写一个 R2 临时文件并发一条 send-queue 消息；文件处理完毕后，不足 1000 条的余量立即写入并入队，不会等待下一次触发。

举例 — 一个包含 2500 条日志的文件：

第 1–1000 条   → processed/xxx-0.txt（1000 条）+ send-queue 消息
第 1001–2000 条 → processed/xxx-1.txt（1000 条）+ send-queue 消息
第 2001–2500 条 → processed/xxx-2.txt（ 500 条）+ send-queue 消息

第四阶段 — Sender Worker  事件驱动，按 max_batch_size / max_batch_timeout 批量聚合

满足以下任一条件时 Worker 被唤醒：

  • max_batch_size = 50：单次调用最多接收 50 条待发送消息
  • max_concurrency 未设置：由 Queue autoscaler 自动扩缩 invocation 数量（当前上限 250）
  • max_batch_timeout = 5s：不足 50 条但已等待 5 秒 → 同样触发

每条消息对应一次独立的 HTTP POST。同一 invocation 内 50 条消息 for-await 串行处理；多个 invocation 之间由 autoscaler 并行调度以维持高吞吐。发送失败由 Queue 重试（最多 5 次），彻底失败后转入 send-dlq。

部署指南
前置条件：本地已安装 Node.js v18+；已安装 Wrangler CLI（npm install -g wrangler）；已准备好 Cloudflare API Token，权限范围限定为目标账号，需包含以下三项：Workers Scripts:Edit、Workers R2 Storage:Edit、Workers Queue:Edit（可使用 Edit Cloudflare Workers 模板后手动追加后两项）。
根据实际部署修改 wrangler.toml：
  • name → Worker 名字（自定义，任意唯一名称）
  • account_id → 必须改为你自己的 Cloudflare Account ID（Dashboard 右上角，或 wrangler whoami 获取）
  • bucket_name → 你的 R2 bucket 名字（在步骤1创建）
  • [vars] 里的 R2_BUCKET_NAME → 必须与上方 bucket_name 完全一致
  • Queue 名字（producers / consumers 中的 queue = "..."）→ 你的实际 Queue 名字（在步骤2创建）
  • [vars] 里的 PARSE_QUEUE_NAME / SEND_QUEUE_NAME → 必须与 queue 名字一致
仓库中 wrangler.toml 包含示例值，部署前须将 account_id 改为你自己的；其他字段按需自定义。
部署前请向日志对接方确认：推送服务器地址（Endpoint）、鉴权私钥（Private Key）、接收 URI 路径（URI Path）。

配置 Logpush Job

Dashboard → [目标域名] → Analytics & Logs → Logpush → Create a Logpush job

  1. 目的地：选 R2
  2. Bucket：cdn-logs-raw  ·  Path prefix：logs/{DATE}/
  3. 数据集：HTTP requests
  4. 字段：全选所有字段（代码对缺失字段有容错处理）
  5. Timestamp format：Unix
  6. Sample rate：1（100% 采样）
Timestamp format 必须选 Unix，否则时间戳解析可能出错。
6b

开启 Content-Length 日志采集 — 可选

默认情况下，字段 #21（sent_http_content_length）回退使用 EdgeResponseBodyBytes。如需采集真实的 Content-Length 响应头值，需通过 API 配置 Logpush Custom Fields。这是一次性的 Zone 级别配置。

此配置只能通过 API 完成，Cloudflare Dashboard UI 不支持设置 raw response fields。

步骤 A — 查找是否已有 Custom Fields Ruleset

# 列出所有 ruleset，找 http_log_custom_fields 阶段
curl "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/rulesets" \
  --request GET \
  --header "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  | jq '.result[] | select(.phase == "http_log_custom_fields") | {id, phase}'

若有返回，记录 id 作为 RULESET_ID，跳至步骤 C。若无返回，执行步骤 B。

步骤 B — 创建 Custom Fields Ruleset（仅步骤 A 无结果时执行）

curl "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/rulesets" \
  --request POST \
  --header "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  --json '{
    "name": "Zone-level phase entry point",
    "kind": "zone",
    "description": "Custom log fields for Logpush",
    "phase": "http_log_custom_fields"
  }' | jq '.result.id'

记录返回的 id 作为 RULESET_ID。

步骤 C — 配置采集 content-length 响应头

使用 raw_response_fields 捕获源站返回的原始 content-length 值（CF Transform Rules 修改之前）。

curl "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/rulesets/$RULESET_ID" \
  --request PUT \
  --header "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  --json '{
    "rules": [
      {
        "action": "log_custom_field",
        "expression": "true",
        "description": "Capture content-length response header for Logpush",
        "action_parameters": {
          "raw_response_fields": [
            { "name": "content-length" }
          ]
        }
      }
    ]
  }'

步骤 D — 在 Logpush Job 中加入 ResponseHeaders 字段

# 先获取 Logpush Job ID
curl "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/logpush/jobs" \
  --request GET \
  --header "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  | jq '.result[] | {id, name, dataset}'

# 更新 Job，加入 ResponseHeaders 字段
curl "https://api.cloudflare.com/client/v4/zones/$ZONE_ID/logpush/jobs/$JOB_ID" \
  --request PUT \
  --header "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  --json '{
    "output_options": {
      "field_names": [
        "CacheCacheStatus","ClientCountry","ClientIP","ClientRequestBytes",
        "ClientRequestHost","ClientRequestMethod","ClientRequestProtocol",
        "ClientRequestReferer","ClientRequestScheme","ClientRequestURI",
        "ClientRequestUserAgent","ClientSSLProtocol","ClientSrcPort",
        "EdgeColoCode","EdgeResponseBodyBytes","EdgeResponseBytes",
        "EdgeResponseContentType","EdgeResponseStatus","EdgeServerIP",
        "EdgeStartTimestamp","EdgeEndTimestamp","EdgeTimeToFirstByteMs","OriginIP",
        "OriginRequestHeaderSendDurationMs","OriginResponseDurationMs",
        "OriginResponseHeaderReceiveDurationMs","OriginResponseStatus",
        "OriginTLSHandshakeDurationMs","ParentRayID","RayID",
        "ResponseHeaders"
      ],
      "timestamp_format": "unix"
    }
  }'
配置完成后，每条日志的 ResponseHeaders 字段将包含 {"content-length": "12345"}，Worker 自动将其用于字段 #21。
注意：HTTP/2 响应或 CF 开启 gzip 压缩时，源站可能不发送 Content-Length 头（改用 chunked transfer encoding），此时字段 #21 输出 -。

环境变量说明

变量名类型示例值说明
CTYUN_ENDPOINTSecret http://log.example.com:5580 日志接收服务器地址，不含 URI 路径，不含末尾斜杠
CTYUN_PRIVATE_KEYSecret YourKey@1234 MD5 鉴权私钥，由对接方提供
CTYUN_URI_EDGESecret /logpost/yourpath HTTP POST 目标 URI 路径，由对接方提供
R2_BUCKET_NAMEVar cdn-logs-raw R2 bucket 名字。必须与上方 [[r2_buckets]] bucket_name 完全一致。
BATCH_SIZEVar 1000 每批 POST 包含的日志行数。Worker 会校验该值必须是 1 到 2000 的整数。
RAW_LOG_PREFIX / RAW_LOG_SUFFIXVar logs/ / .log.gz 原始 Logpush 对象 allowlist。不匹配的 R2 通知会被 ack 并跳过。
LOG_LEVELVar info Worker 日志级别。排查问题时改为 debug，生产恢复 info
PUSH_START_TIMEVar 空 统一时间点控制 — Worker 自动识别未来/过去时间。
默认为空，不过滤，所有日志正常处理推送。
未来时间：Parser 自然等待，该时间点之前的日志静默跳过。
过去时间（补救模式）：部署后 Cron 触发器（每分钟）约 1 分钟内自动触发补救：扫描 R2 RAW_LOG_PREFIX/YYYYMMDD/ 目录（默认 logs/YYYYMMDD/），把该时间点到当前时刻的所有历史文件重新入队到 parse-queue。补救执行中使用 .recover-running-* 临时标记，只有全部匹配文件成功入队后才写入 .recover-done-*。
格式：ISO 8601，例如 2026-04-15T10:00:00Z（UTC）或 2026-04-15T18:00:00+08:00（北京时间）。
自动失效：所有新文件时间均满足条件后，过滤逻辑永远不触发，无性能损耗。
Secret 通过 wrangler secret put <NAME> 一次性设置后持久保存在 Cloudflare 云端，任何情况下不得写入 wrangler.toml 或提交至代码仓库。

指标解读指南

信号含义处理方式
Backlog = 0 + Lag < 30s系统实时消化，运行健康正常，无需操作
Backlog 持续增长消费速度跟不上生产速度，真正的瓶颈信号检查 Worker 错误日志；必要时扩容
CPU Time Exceeded > 0单个 Logpush 文件过大，单次调用 CPU 超限持续监控频率；不要降低 max_batch_size
Messages Retried > 0接收端不稳定排查 5xx 子请求；确认接收端容量
DLQ > 0有数据丢失风险，最高优先级立即检查 DLQ 消息；从 R2 手动补传
send-queue 积压持续增长接收端变慢或不可用，或 send-queue max_concurrency 被人为压低检查接收端延迟和错误率；确认 max_concurrency 未被压低（推荐留空 autoscale）

紧急补救：补传历史日志

如需从过去的某个时间点开始推送日志（例如：客户原有日志管道故障，需要 CF 侧补传），只需将 PUSH_START_TIME 设置为该过去时间。Worker 的 Scheduled 处理器（Cron 每分钟触发）会自动检测并执行一次性补救：

# 示例：客户管道在今天 15:00 出故障，16:00 发现并决定由 CF 补传
# 需求：补传 15:00–16:00 历史日志 + 继续推送新日志

# 1. 编辑 wrangler.toml
PUSH_START_TIME = "2026-04-22T15:00:00+08:00"

# 2. 推送 — GitHub Actions 约 1 分钟内完成部署
git add wrangler.toml
git commit -m "recovery: backfill from 15:00"
git push

# 3. 等待约 1 分钟，Cron 检测到过去时间，自动执行：
#    - 写入临时标记 .recover-running-<时间戳> 到 R2
#    - 扫描 R2 RAW_LOG_PREFIX/YYYYMMDD/（默认 logs/YYYYMMDD/）下 [15:00, 当前] 时间范围的文件
#    - 批量入队到 parse-queue（每批 100 条消息）
#    - Parser 正常按 PUSH_START_TIME 过滤并处理
#    - 历史日志流向接收端
#    - 新日志继续通过 R2 Event Notification 正常推送
#    - 全部入队成功后才写入 .recover-done-<时间戳>
幂等设计：只有所有匹配文件成功入队后才写入 .recover-done-<编码时间戳>；如果补救中途失败，临时标记会被清理，下一次 Cron 自动重试。
若将 PUSH_START_TIME 改为不同的过去时间，会触发新的补救。重叠入队由确定性 batch key、Sender 同 batch 内串行 和 .done 标记处理，不假设客户侧改动。
Cron 触发器每分钟运行，无补救任务时零开销（约 0.1ms）。实际执行补救时成本：R2 list + 每 100 条一次 Queue 批量发送，根据文件数量在数秒到数分钟内完成。
